# 베이스라인 프롬프트 (P0 · P1 · P2)

출제위원용 참조 프롬프트. 프롬프트 엔지니어링 기법을 한 단계씩 쌓아 성능 변화를 실측한다.

| 파일 | 추가 기법 | 설명 |
|---|---|---|
| `P0.txt` | (없음) | 최소 지시 — 예측 작업 한 문장만 (롤·출력 형식 미지정). |
| `P1.txt` | **출력 형식·허용 라벨 명시** | 예측 작업 + 허용 라벨과 `risk_grade, cycle_range` 한 줄 형식·예시. |
| `P2.txt` | **롤 + 판단 신호 힌트 + 형식** | P1에 롤("K-511 정비 전문가")과 핵심 판단 신호(1년 정비횟수·누적 가동시간·직전고장 경과일·부품 사용 이력) 힌트를 얹은 종합형. |

> 롤(페르소나)만 추가한 이전 P1은 최소 지시 P0과 성능 차이가 거의 없어 제거하고, 형식·힌트 단계를 앞당겼다.

## 문제: K-511 계통별 이중 라벨

각 입력 행(K-511 부품 계통별 정비 이력 1건)에 대해 **두 라벨**을 예측한다:
- `risk_grade` ∈ {HIGH, MEDIUM, LOW}
- `cycle_range` ∈ {0-30, 31-90, 91-180, 181+}

출력은 행당 `risk_grade, cycle_range` 콤마 한 줄 (예: `HIGH, 0-30`). 채점은 두 축을 각각
Macro F1 으로 계산한 뒤 평균한다.

## 사다리 설계 (형식·힌트 레이어링)

세 단은 **출력 형식 / 롤 / 판단 힌트**를 한 겹씩 얹어 각 요소의 기여를 분리 측정한다
(P0=최소, P1=형식, P2=롤+형식+힌트). few-shot·CoT는 의도적으로 뺐다 — 이전 발전기
프로토타입(단일 라벨)에서 무캘리브레이션 few-shot·CoT가 over-flagging 편향으로 다수 클래스를
붕괴시켜 Macro F1을 오히려 낮췄기 때문. K-511 이중 라벨에서 few-shot 재도입은 별도 과제로 둔다.

## 설계 원칙

- **프롬프트 파일에는 주석/메타 금지** — 파일 내용 그대로 LLM system 메시지로 들어간다.
- **출력은 행당 한 줄 `risk_grade, cycle_range`** — 하네스의 출력 토큰 캡 50(예: `MEDIUM, 91-180`
  수용) + 정답-only 채점과 정합.

## 라벨 중복 주의 (운영자 참고)

data/ 실측 1,000/1,000행에서 `cycle_range` 가 `risk_grade` 를 결정한다:
`0-30→HIGH · 31-90→MEDIUM · 91-180→MEDIUM · 181+→LOW`. MEDIUM 내부 분기는
`operation_hours_total ≥ 2700` 로 535/535 완전 설명. 즉 `risk_grade` 는 `cycle_range` 의
결정적 축약이라 결합 Macro F1 평균은 HIGH/LOW 경계를 이중 계산한다(주최측 재논의 flag).
**단, 채점 코드는 두 축을 모델의 리터럴 출력에서 독립 채점하며 한 축에서 다른 축을 추론하지 않는다.**

## 실측

```bash
uv run grader run --prompt baselines/P1.txt --split sample --runs 1   # 연습용(10행)
uv run grader run --prompt baselines/P1.txt --split public --runs 1   # 기준(300행, 유료)
```

> sample(10행)은 표본이 작아 배선·연습 확인용. 프롬프트 우열 비교는 public(300행) 기준.
