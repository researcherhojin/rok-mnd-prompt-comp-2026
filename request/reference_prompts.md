# 참고(레퍼런스) 프롬프트 — ≈50점 · ≈80점 (고객사 전달용)

대회 난이도·점수 분포를 가늠하기 위한 **참고 프롬프트 2종**과 실측 점수. 참가자 제출용이 아니라
**고객사(개발팀) 내부 참고용**이다. 두 프롬프트는 데이터에서 역설계한 **정답 생성 규칙을 인코딩**하므로
**참가자에게 공개 금지**(공개 시 정답 유출).

## ⚠️ 핵심 발견 — 점수 천장은 "글자수"가 아니라 "모델"이 결정

동일한 **≈80점 프롬프트(650자·정답-only)**를 모델만 바꿔 `public`(300행) 실측:

| 모델 | 결합 Macro F1 | 리더보드 점수 |
|---|---|---|
| `gpt-4o-mini` | 0.1766 | **23.73** |
| `gpt-4.1-nano` | 0.3131 | **36.01** |
| `gpt-5.4-nano` | 0.35 | **≈35** |
| `gpt-5.4-mini` | 0.54 | **≈54** |
| `gpt-5.4` | 0.81 | **≈80** |

> `gpt-4o-mini`는 `gpt-5.4-mini`와 **이름만 비슷할 뿐 결과가 전혀 다르다**(0.1766 vs 0.54).
> 모델명의 `mini`/`nano` 표기로 성능을 유추하지 말 것 — 아래 §"`gpt-4o-mini` 실측" 참조.

- **글자수 제한과 무관하다.** 규칙을 다 담은 650자 프롬프트도 `gpt-5.4-nano`에서는 ~35점이 한계다.
  오프라인에서 같은 규칙(결정트리)은 결합 F1 **0.82**를 내므로, **규칙이 부족한 게 아니라 소형 모델이
  `max_tokens=50`·정답-only 제약에서 다분기 수치 규칙을 실행하지 못한다.**
- 즉 **"80점짜리 프롬프트가 존재하려면" 충분히 강한 모델(gpt-5.4급)이 필요**하다. non-reasoning 소형
  (nano)로 고정하면 완벽한 규칙 프롬프트도 ~35점 이상 나오기 어렵다 → **모델 선택이 대회 점수 분포·
  변별력을 좌우**한다(주최측 결정 사항).

## `gpt-4o-mini` 실측 — 채점 신호가 **역전**된다 (split `public` 300행, temperature 0, seed 42)

| 프롬프트 | 결합 Macro F1 | 리더보드 점수 | 완전일치 | 글자수 |
|---|---|---|---|---|
| ≈80점 (full 규칙, depth-4) | **0.1766** | 23.73 | 81/300 | 650 |
| ≈50점 (coarse 규칙, 3분기) | **0.3102** | 36.51 | 124/300 | 422 |
| *(대조)* 다수클래스 상수 예측 | 0.1711 | — | — | — |

INVALID 0행 — 출력 형식은 정상이며, 틀린 것은 판정 내용이다.

- **상수 예측기로 붕괴한다.** full 규칙에서 `risk` HIGH의 recall이 0.961인 반면 LOW는 F1 0.000,
  `cycle`도 `181+`가 F1 0.000이다. 즉 대부분의 행을 `HIGH, 0-30`으로 밀어버린다. 그 결과 0.1766은
  아무 규칙 없이 다수클래스만 찍는 **상수 예측(0.1711)과 사실상 구별되지 않는다.**
- **좋은 프롬프트가 오히려 불리하다.** 규칙이 정교한 650자(0.1766)가 조잡한 422자(0.3102)보다
  낮다. depth-4 분기를 주면 실행하지 못하고 무너지기 때문이다. 이 모델로 대회를 열면 채점 점수가
  프롬프트 품질과 **역상관**한다 — 변별력 저하가 아니라 **신호의 부호가 뒤집히는** 문제다.
- 따라서 `gpt-4o-mini`를 채점 모델로 고정하면 §"실측 결과"의 골드셋 앵커(52.1 · 80.3)와
  "≈80점 프롬프트"라는 명칭 자체가 성립하지 않는다. **모델은 미확정으로 두고 주최측 판단이 필요하다.**

## `gpt-4.1-nano` 실측 — 순서는 보존되지만 `max_tokens=50`이 사다리를 깨뜨린다

`gpt-4o-mini`의 대안 후보로 5개 프롬프트를 같은 조건(`public` 300행, temperature 0, seed 42,
전 실행 ERROR 0)에서 측정했다. P0–P2는 `defense-prompt-grader/baselines/`의 품질 사다리
(역할만 → 출력형식 추가 → 도메인 추론 추가)이고, ≈50·≈80은 위 정답 규칙 인코딩 프롬프트다.

| 프롬프트 | 성격 | 글자수 | 결합 F1 | 리더보드 | 완전일치 | INVALID |
|---|---|---|---|---|---|---|
| P0 | 역할 지시만 | 65 | **0.0000** | 9.78 | 0/300 | **1.0000** (300행) |
| P1 | + 라벨·출력형식 | 211 | **0.2337** | 30.33 | 71/300 | 0.0000 |
| P2 | + 도메인 추론 6단계 | 822 | **0.1738** | 22.90 | 38/300 | **0.5033** (151행) |
| ≈50점 | 규칙 coarse | 422 | **0.1903** | 25.72 | 44/300 | 0.0033 (1행) |
| ≈80점 | 규칙 full | 650 | **0.3131** | 36.01 | 62/300 | 0.0000 |

- **`gpt-4o-mini`보다 낫다.** 규칙 프롬프트 간 순서가 보존된다(≈80 0.3131 > ≈50 0.1903).
  `gpt-4o-mini`는 이 순서가 역전됐다(0.1766 < 0.3102). `cycle` 4개 클래스 F1도 각각
  0.376 / 0.333 / 0.297 / 0.293으로 **전 클래스에 분포**한다(`gpt-4o-mini`는 `181+`가 0.000).
- **그러나 사다리는 단조롭지 않다.** 순위가 ≈80 > P1 > ≈50 > P2 > P0 로, P1(211자)이
  P2(822자)를 앞선다.
- **원인은 판정 실패가 아니라 출력 절단이다.** P2의 INVALID 151행 원문은 전부 69~91자에서
  문장 중간에 끊겨 있다(예: `'…긴급 정비는 아니지만 정'`). P2의 "판단 방법 1)~6)"이 단계별
  서술을 유도했고 `max_tokens=50`이 답을 내기 전에 잘랐다. 정상 응답 평균 길이는 11.6자다.
- **따라서 현재 하네스에서 점수를 가르는 지배 요인은 도메인 실력이 아니다.** ① 출력 형식을
  명시했는가(P0 0.0000 → P1 0.2337) ② 장황한 추론을 유도하지 않았는가(P2 < P1). 모델을 바꿔도
  이 구조는 남는다 → **`max_tokens` 상향 또는 고정 시스템 지시의 출력 형식 강제를 검토해야 한다.**
  (현재 시뮬레이터는 user 메시지에 데이터만 넣고 출력 형식을 강제하지 않아, 참가자 지침이
  시스템 프롬프트 전부다. P0의 INVALID 100%가 그 결과다.)

### 처방 검정 — `max_tokens` 상향은 해법이 아니고, 고정 시스템 지시가 해법이다

절단이 원인이라면 캡을 올리면 낫는가? **아니다.** 반대로 고정 시스템 지시로 출력 형식을 강제하면
캡을 그대로 두고도 해결된다. 둘을 직접 검정했다(모두 `gpt-4.1-nano` · `public` 300행).

| 조건 | P2 결합 F1 | P2 INVALID |
|---|---|---|
| `max_tokens=50` (현행) | 0.1738 | 0.5033 |
| `max_tokens=200` (상향) | 0.1813 | 0.3967 |
| `max_tokens=50` + **고정 시스템 지시** | **0.2741** | **0.0267** |

- **캡 상향은 실패했다.** 200으로 올려도 남은 INVALID 119행이 **119/119 전부 캡 근처(300~350자)에서
  또 절단**됐다. 모델이 캡만큼 더 길게 늘어놓을 뿐 답에 도달하지 않는다. 캡을 키우는 만큼 비용·지연만
  늘고 문제는 남는다.
- **고정 시스템 지시가 해결한다.** 참가자 지침 뒤에 "추론 과정을 쓰지 말고 `risk_grade, cycle_range`
  한 줄만 출력" 문장을 덧붙이자, `max_tokens`는 50 그대로인데 INVALID가 **0.5033 → 0.0267**로
  떨어지고 F1이 **0.1738 → 0.2741**로 올랐다.

**같은 조건(고정 시스템 지시 + `max_tokens=50`)에서 사다리가 복원된다:**

| 프롬프트 | 지시 없음 | **고정 지시 적용** | INVALID(적용 후) |
|---|---|---|---|
| P0 (역할만) | 0.0000 | **0.1675** | 0.7867 |
| P1 (+출력형식) | 0.2337 | **0.2693** | 0.0000 |
| P2 (+도메인 추론) | 0.1738 | **0.2741** | 0.0267 |
| ≈50점 (규칙 coarse) | 0.1903 | **0.2024** | 0.0133 |
| ≈80점 (규칙 full) | 0.3131 | **0.3004** | 0.0000 |

- **P0 < P1 < P2 가 단조 증가로 회복된다**(0.1675 < 0.2693 < 0.2741). 지시가 없을 때는 P2 < P1로
  역전돼 있었다. ≈50 < ≈80 순서도 유지된다.
- P0가 여전히 INVALID 0.7867인 것은 **정당한 실패**다. P0는 라벨 어휘(HIGH/MEDIUM/LOW, 4개 구간)를
  알려주지 않으므로 형식만 강제해도 값이 틀린다 — 그 어휘를 명시하는 것이 P1의 기여이고, 대회가
  변별해야 할 지점이다.
- ⚠️ **단, P1과 P2의 격차는 0.0048로 얇다.** 1회 실행이라 변동성 추정치가 없다. 배점·변별력을
  확정하기 전에 `--runs 2` 이상으로 재확인이 필요하다.

> **결론:** 설계 문서의 6단계 하네스 1단계 **"고정 시스템 지시 구성"을 실제로 구현해야 한다.**
> 현재 시뮬레이터는 이 단계가 없어 참가자 지침이 시스템 프롬프트 전부이며, 그 결과 출력 형식
> 준수 여부가 도메인 실력을 압도한다. `max_tokens` 상향은 이 문제의 해법이 아니다.

> **⚠️ 캐시 키 주의(구현 참고):** `cache_key`는 `prompt + model + split`만 해싱하므로
> `max_tokens`·`temperature`·`seed`를 바꿔도 **이전 응답이 그대로 재사용된다.** 위 실험은 별도
> DB 파일과 `force=True`로 격리해 수행했다. 파라미터 실험 시 반드시 캐시를 분리할 것.

> **⚠️ 공정성 결함(측정 중 발견):** LLM 호출 실패 셀은 예측값 `ERROR`로 격리돼 **오답으로 계상**된다.
> 실행 중단 임계는 `max(1, int(0.05 × 총셀수))`이므로 `public` 300행에서는 15셀이다. P2 최초
> 실행에서 13셀이 실패했고 문턱 아래라 중단 없이 완주해 F1이 0.1697로 나왔다(재실행 후 클린값
> **0.1738**). 참가자는 일시적 API 오류로 점수를 잃고도 이를 알 방법이 없다. 운영 시 **실패 셀
> 재시도** 또는 최소한 **제출 상세에 실패 셀 수 노출**이 필요하다.

## 실측 결과 (모델 `gpt-5.4`, split `public` 300행, temperature 0, seed 42)

| 프롬프트 | 리더보드 점수 | 결합 F1 | 완전일치 | 글자수 |
|---|---|---|---|---|
| ≈50점 (coarse 규칙) | **52.1** | 0.483 | 191/300 | 422 |
| ≈80점 (full 규칙) | **80.3** | 0.805 | 244/300 | 650 |

> 리더보드 점수 = 0.9 · 예측 성능(결합 Macro F1) + 0.1 · 효율성`((3000−글자수)/3000)`.
> `gpt-5.4`는 규칙을 거의 오프라인 상한대로 실행(측정 F1 ≈ 규칙 F1 − 0.01~0.02).

## 근거 규칙 (데이터 1,000행 역설계)

`cycle_range`가 `risk_grade`를 결정하므로 **cycle_range만 맞추면 된다**. 핵심 변수는
**누적 가동시간(H)·최근 1년 정비횟수(M)·직전고장 경과일(F)·부품 정비후 가동시간(S)**. 결정트리
depth-4가 결합 F1 0.82, depth-6이 0.90. `H ≤ 2700` 분기는 설계 문서(§1.3)의 MEDIUM 분기와 일치.

## ≈50점 프롬프트 (coarse — 3분기)

```
당신은 K-511 트럭의 부품 계통별 정비 판정 엔진입니다. 각 행마다 아래 규칙을 위에서부터 순서대로 적용해 cycle_range를 정하고 risk_grade를 붙여 "risk_grade, cycle_range" 한 줄만 출력하세요. 다른 설명은 쓰지 마세요.

사용할 값: H = 차량 누적 가동시간(시간), M = 최근 1년 정비 횟수(회), F = 부품 직전고장 경과일(일).

cycle_range 결정:
- F ≤ 25 이면 → 0-30
- 아니고 H ≤ 2700 이면: M ≤ 3 이면 91-180, 아니면 0-30
- 아니면(H > 2700) → 31-90

risk_grade는 cycle_range로 결정: 0-30 → HIGH, 31-90 → MEDIUM, 91-180 → MEDIUM, 181+ → LOW.
예: HIGH, 0-30
```

## ≈80점 프롬프트 (full — depth-4 규칙)

```
당신은 K-511 트럭의 부품 계통별 정비 판정 엔진입니다. 각 입력 행에 대해 아래 규칙을 순서대로 적용해 먼저 cycle_range를 정하고, 그 값으로 risk_grade를 정하세요. 설명 없이 "risk_grade, cycle_range" 한 줄만 출력합니다.

사용할 값: H = 차량 누적 가동시간(시간), M = 최근 1년 정비 횟수(회), F = 부품 직전고장 경과일(일), S = 부품 정비 후 가동시간(시간).

cycle_range 결정:
- H ≤ 2700 인 경우:
  - M ≤ 3: F ≤ 25 이면 0-30 / 25 < F ≤ 90 이면 91-180 / F > 90 이면 (S ≤ 820 이면 181+, 아니면 91-180)
  - M = 4: S ≤ 922 이면 91-180, 아니면 0-30
  - M ≥ 5: 0-30
- H > 2700 인 경우:
  - H ≤ 4700: M ≤ 3 이면 (F ≤ 25 이면 0-30, 아니면 31-90) / M ≥ 4 이면 (F ≤ 334 이면 0-30, 아니면 31-90)
  - H > 4700: 0-30

risk_grade는 cycle_range로 결정: 0-30 → HIGH, 31-90 → MEDIUM, 91-180 → MEDIUM, 181+ → LOW.
예: HIGH, 0-30
```

## 재현 방법

```bash
# 강한 모델을 .env에 설정 (LLM_MODEL=gpt-5.4) 후:
uv run grader run --prompt <prompt.txt> --split public --runs 1 --model gpt-5.4
```

- 점수를 더 높이려면 depth-6 규칙(부품 정비후 주행거리·부품 장착 경과 개월 추가)까지 인코딩(오프라인 0.90).
- 소형 모델(nano)로 고정할 경우 이 프롬프트들도 ~35점대로 내려간다 — 위 §"핵심 발견" 참조.
