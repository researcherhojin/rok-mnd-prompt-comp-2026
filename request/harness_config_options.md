# 하네스 커스터마이즈 항목 (개발팀 전달용)

이 문서는 재실행 채점 하네스(`defense-prompt-grader`)에서 **주최측이 조정 가능한 파라미터**를
한곳에 정리한 전달용 명세입니다. 표의 "현재값"은 코드 기본값 실측이며, 대부분 환경변수(`.env`)로
오버라이드하거나 설정 상수로 관리합니다. "상태" 열의 **확정**은 값만 바꾸면 되는 항목,
**미확정(주최측)**은 대회 의도·문제 수 확정 후 결정할 항목입니다.

> 파라미터를 바꾼 뒤에는 반드시 재검증(`uv run pytest`)과 baseline 재측정을 거칩니다.
> 배점·글자수 등 "미확정" 항목은 값이 정해지기 전까지 회귀 앵커로 고정하지 않습니다.

---

## 1. 모델 · 실행 (LLM)

| 항목 | 현재값 | 설정 위치 | 조정 범위 · 제약 | 상태 |
|---|---|---|---|---|
| 채점 모델 | **미정** | `LLM_MODEL` (env) | ⚠️ `gpt-4o-mini`는 실측에서 상수 예측기로 붕괴(아래 참조) — 소형이면 무엇이든 된다고 가정하지 말 것. 후보별 실측 필수 | 미확정(실측 후 확정) |
| temperature | `0` | `LLMConfig` | 재현성 위해 0 고정 권장 | 확정 |
| top_p | `1.0` | `LLMConfig` | 0 온도와 함께 결정성 확보 | 확정 |
| max_tokens | `50` | `LLMConfig` | "위험등급, 정비주기" 한 줄 출력에 맞춤. 키우면 비용·지연 증가 | 확정(출력 계약과 정합) |
| seed | `42` | `LLM_SEED` (env) | **재현성 고정용. 참가자에게 비노출** (UI엔 `temperature 0`만 표시). 값 자체는 자유 | 확정 |
| 동시 호출 수 | `8` | `MAX_CONCURRENCY` (env) | 서버보다 **API 세션 점유가 병목** → 배치·큐로 조절 | 운영값(환경별 조정) |
| 재시도 | 최초 1 + 3회 | `llm.py` 상수 | 429/5xx/네트워크 오류만 재시도. `Retry-After` 상한 20초 | 확정 |

> **출력 토큰 파라미터**: 최신 OpenAI 모델(gpt-5.x 등)은 `max_completion_tokens`를 요구하고 구형 `max_tokens`는 400을 낸다. 하네스가 `max_completion_tokens`를 먼저 쓰고 미지원 시 `max_tokens`로 자동 폴백한다. 모든 4xx는 **응답 본문을 로깅**(uvicorn 콘솔)하고 실패 사유를 리더보드/제출 `error_msg`에 노출한다. `>5%` 셀 실패 시 전체 실행을 중단한다.
> **표시 연동**: 참가자 제출 화면·운영 페이지의 모델/temperature/글자수 하드캡은 하드코딩이 아니라 `GET /api/config`로 서버 설정을 받아 표시한다(seed는 비노출).
> **모델별 점수 천장(실측·중요)**: 동일한 규칙 인코딩 프롬프트(650자, 정답-only)가 `gpt-4o-mini` **0.1766** / `gpt-5.4-nano` **≈35** / `gpt-5.4-mini` **≈54** / `gpt-5.4` **≈80**. 오프라인 규칙 상한은 0.82인데 소형 모델은 `max_tokens=50`·정답-only에서 다분기 규칙을 실행 못 해 완벽한 규칙을 줘도 상한이 낮다. **모델 선택이 대회 점수 분포·변별력을 좌우**한다(주최측 결정). 참고 프롬프트·실측 → `reference_prompts.md`.
>
> ⚠️ **`gpt-4o-mini` 실측 — 채점 신호가 역전된다**: 결합 F1 **0.1766**(리더보드 23.73)은 다수클래스 상수 예측 **0.1711**과 사실상 구별되지 않는다(`risk` HIGH recall 0.961 / LOW F1 0.000 — 대부분을 `HIGH, 0-30`으로 밀어버림). 게다가 조잡한 422자 프롬프트가 **0.3102**로 정교한 650자를 앞선다. 즉 이 모델을 고정하면 점수가 프롬프트 품질과 **역상관**한다. 모델명의 `mini`/`nano` 표기로 성능을 유추하지 말 것(`gpt-4o-mini` 0.1766 vs `gpt-5.4-mini` 0.54). 후보 모델은 반드시 `public` 300행 실측 후 채택한다.

## 5. 운영 · 인프라 (개발팀 스택)

- **모바일 우선 + 세션 지속**: 중단·재개, 콘솔 닫아도 서버 저장·백그라운드 진행.
- **제출 흐름**: 제출 즉시 접수 확인 → 백그라운드 채점 → 수 분 후 리더보드 반영(자정 배치 완화, 군 취침시간 고려).
- **부정 방지**: 문제 붙여넣기 즉답 방지 함정 설계 + 비-DS 지인 pool adversarial(off-topic/jailbreak) QA.
- 재현성·감사: raw 출력 / 파싱 / 점수를 저장(LLM 출력이 완전 결정적이지 않음).
- **좀비 작업 복구(필수)**: 채점 워커가 죽으면(배포·리로드·크래시·API 쿼터 소진) `running` 제출이 그대로 남아 프론트가 무한 폴링한다(시뮬레이터 실측: 서버 자동 리로드로 in-flight 작업이 orphaned → DB엔 `running`인데 워커 없음). 운영 플랫폼은 ① 작업 heartbeat·타임아웃(N분 무진전 → `failed` 자동 전환) ② 배포/재기동 시 in-flight 작업 재적재(re-enqueue) 또는 startup 시 `running`→`failed` 재조정(reconcile) ③ 클라이언트 폴링 상한을 갖춰야 한다. 시뮬레이터는 큐/재적재가 없는 단일 프로세스라 이 갭을 명세로만 전달한다.

## 6. 참가자 플랫폼 (리더보드 · 게시판) 명세 — 개발팀 구축 대상

시뮬레이터 콘솔은 **리더보드**를 클라이언트 집계로 시연한다. **게시판은 콘솔에서 제외**(운영
플랫폼 몫)이며 아래 §6.2 명세만 개발팀에 전달한다. 실 운영 플랫폼(데이원컴퍼니)에서는 둘 다
서버측으로 구현한다.

### 6.1 리더보드 집계 (콘솔은 클라이언트 계산, 운영은 서버 권장)

| 항목 | 규칙 | 비고 |
|---|---|---|
| 최고점 자동 선택 | 참가자(name)별 done 제출 중 최고 `leaderboard_score` 1건을 대표로 순위 | 나머지는 누적 수로만 집계 |
| 누적 제출 | name별 전체 제출 수 | 1일 3회 한도(운영 정책) |
| 최종 제출 | name별 최신 `created_at` | |
| 소속 | 제출 시 입력(`affiliation`, 선택). 리더보드 표시 | `submissions.affiliation TEXT` 컬럼 |
| 이름 마스킹 | 가운데 `○` (홍길동→홍○동). 공개 토글 | 개인정보 보호 |
| 내 위치 | 본인 순위·백분위(상위 %)·점수 구간 히스토그램 강조 | |
| 정렬 | 점수 내림차순, 동점은 최초 제출 우선 | 최종 동점 규칙 §4 |

- **참가자 식별**: 현 콘솔은 `name` 문자열뿐. 운영은 군 SSO 등 안정적 식별자 권장(소속도 SSO 연동 가능).
- **반영 지연**: 제출 즉시 접수 → 백그라운드 채점 → 수 분 후 리더보드 반영(군 취침시간 고려, 자정 배치 완화).

### 6.2 게시판 (공지 · Q&A) — 전면 신규 (콘솔 미포함)

콘솔에는 없다(운영 플랫폼 전면 구축). 구축 항목:

- **공지(announcements)**: 제목·본문·고정(pinned)·작성시각. 운영자만 작성.
- **Q&A(posts + answers)**: 참가자 질문 + 운영팀 답변. 인증(참가자/운영자 역할), 모더레이션(신고·숨김),
  정답·채점 로직 유출 방지 가이드.
- **테이블(예)**: `announcements(id, title, body, pinned, created_at)`,
  `qna(id, author, body, created_at, status)`, `qna_answers(id, qna_id, author_role, body, created_at)`.
- **라우트(예)**: `GET/POST /api/announcements`, `GET/POST /api/qna`, `POST /api/qna/{id}/answer`.
- 인증·권한·스팸 방지는 운영 플랫폼 표준을 따른다(시뮬레이터 범위 밖).

## 7. 주최측 확정 필요 항목 (요약)

아래는 값 변경이 아니라 **대회 설계 결정**이 필요한 열린 항목입니다(각 문서 §7/§11 참조).

1. 대회 의도(능력 vs 경험) · 1 vs 4 문제 채점 구조
2. 리더보드 점수 / 토탈 점수 배점 (§4 가중치)
3. 제출 형태(단일 Shape A vs 가이드형 다단계 B)
4. `sample` 건수 증량 여부
5. 글자수 하드캡 — 3,000자(1,500 대안 논의 있었음). 추가 조정 시 주최측
6. `risk_grade` ↔ `cycle_range` 라벨 중복(데이터 재설계) — 하네스는 두 축 독립 채점, 문서에 flag
